# 자취생 절약 대시보드 설계

## 목표
- GitHub Pages 기반 사이트를 `자취생의 식비·생활비 절약 대시보드`로 재정의한다.
- 공개 핫딜 수집 구조와 Google 로그인 흐름은 유지하면서, 자취생이 `지금 사두면 생활비를 아낄 수 있는 물건`을 빠르게 찾게 만든다.
- 키워드 알림은 브라우저 알림에 더해 로그인한 Google 메일로 즉시 발송되도록 확장한다.

## 핵심 사용자 경험
- 첫 화면에서 `지금 아끼기 좋은 딜`, `식비 절약 딜`, `생활비 절약 딜`, `최근 올라온 핫딜`을 바로 본다.
- 제목만 봐도 어느 플랫폼 상품인지 알 수 있어야 하므로 모든 제목은 `[플랫폼] 상품명` 형식으로 통일한다.
- 태그는 설명어 없이 `야채·과일 / 네이버쇼핑 / 개드립`처럼 값만 노출한다.
- `홈` 버튼과 사이트 제목은 모두 같은 동작을 하며, 클릭 시 검색·소스·카테고리·정렬을 초기화하고 상세 모달을 닫고 최상단으로 이동한다.
- 모바일에서도 한 손으로 탐색 가능해야 하므로 카테고리 탭은 가로 스크롤 칩, 카드 액션은 큰 버튼, 상세 패널은 화면 폭 기준으로 재배치한다.

## 메인 카피
- 메인 문구는 아래 문장을 그대로 사용한다.

> 세상 모든 할인과 핫딜을 빠르게 확인하고, 자취생의 생활비를 아껴주는 특가를 놓치지 마세요.

## 정보 구조

### 상단 헤더
- 좌측 또는 상단 첫 줄에 `홈` 버튼을 배치한다.
- 서비스 제목은 `핫딜모음`을 유지하고, 클릭 시 홈으로 이동한다.
- Google 로그인, 로그아웃, 스크랩, 키워드 알림 흐름은 기존 구조를 유지한다.
- `Firebase 설정 전: 데모 모드` 문구는 제거한다.
- `동기화 상태: 로컬 모드` 대신 `마지막 핫딜 업데이트: YYYY. M. D. 오전/오후 h:mm`를 `data/deals.json.generatedAt` 기준으로 표시한다.

### 홈 대시보드 블록
- `지금 아끼기 좋은 딜`
  - 최근성, 생활밀착도, 가격 명확성, 구매 링크 유무를 기준으로 상위 딜을 보여준다.
- `식비 절약 딜`
  - `야채·과일`, `냉동식품`, `디저트`, `기타 식품` 카테고리만 모아 보여준다.
- `생활비 절약 딜`
  - `생활용품`, `청소용품`, `상품권`, `할인페스타` 중심으로 보여준다.
- `최근 올라온 핫딜`
  - 전체 리스트와 필터 UI를 유지한다.

## 카테고리 체계
- `전체`
- `야채·과일`
- `냉동식품`
- `디저트`
- `기타 식품`
- `생활용품`
- `청소용품`
- `여행`
- `상품권`
- `게임`
- `전자기기`
- `해외핫딜`
- `할인페스타`

### 카테고리 분류 규칙
- 해외 쇼핑몰 또는 해외 직구 성격이 분명하면 품목보다 `해외핫딜`을 우선한다.
- 식품은 다음 순서로 세분화한다.
  - 채소, 과일, 샐러드, 신선 농산물은 `야채·과일`
  - 냉동만두, 냉동치킨, 냉동도시락, 냉동간편식은 `냉동식품`
  - 쿠키, 초콜릿, 아이스크림, 베이커리, 음료 디저트류는 `디저트`
  - 위 세 가지에 속하지 않는 정육, 수산, 유제품, 곡물, 반찬, 일반 식품, 건강식품, 배달앱 식품딜은 `기타 식품`
- 세제, 휴지, 수납, 욕실, 주방 소모품은 `생활용품`
- 청소포, 청소기 소모품, 곰팡이 제거제, 세정제는 `청소용품`
- 숙박, 교통, 항공, 렌터카, 여행 패스는 `여행`
- 기프티콘, 상품권, 포인트 전환, 페이 적립은 `상품권`
- PC 부품, 콘솔, 게임 패키지, 스토어 게임 할인은 `게임`
- 모니터, 노트북, 휴대폰, 가전, 이어폰, 배터리 등은 `전자기기`
- 카드사 행사, 멤버십 행사, 앱쿠폰, 대형몰 시즌전, 브랜드 프로모션은 `할인페스타`

## 카드 및 상세 보기 규칙

### 제목
- 모든 딜 제목은 `[플랫폼] 상품명`으로 정규화한다.
- 플랫폼을 찾을 수 없는 경우에만 `[미확인] 상품명` 같은 임시값이 아니라, 가능한 범위에서 원문 쇼핑몰 또는 소스 분석값을 우선 사용한다.

### 태그
- 카테고리, 플랫폼, 출처를 `값만` 보여준다.
- 예시: `기타 식품 / 네이버쇼핑 / 개드립`
- `플랫폼`, `출처` 같은 접두어는 노출하지 않는다.

### 가격 표기
- 원문 통화를 유지한다.
- 한국 원화는 `9,800원`
- 미국 달러는 `$7.96`
- 기타 통화도 원문 표기를 최대한 유지한다.

### 상태 표시
- 실제 마감 시한 있음 → `남은 시간` 표시
- 마감 시한은 없지만 `선착순`, `한정수량`, `오늘만` 같은 조건 있음 → `선착순` 표시
- 둘 다 없음 → 아무것도 표시하지 않음
- 존재하지 않는 마감 정보를 추정해서 넣지 않는다.

### 상세 보기
- 구매 링크가 있으면 대표 CTA는 `구매하러 가기`
- 실제 마감 시한이 있을 때만 `남은 시간` 카드 노출
- 없고 조건만 있으면 `선착순` 배지 또는 요약 포인트만 노출
- 구매 링크가 없으면 잘못된 커뮤니티 원문으로 보내지지 않도록 대표 CTA는 숨긴다.

## 모바일 설계
- 헤더 버튼은 두 줄까지 자연스럽게 줄바꿈한다.
- 홈/로그인/알림 버튼은 터치 영역 44px 이상을 확보한다.
- 카테고리 탭은 가로 스크롤 가능한 칩 리스트로 바꾼다.
- 필터 영역은 모바일에서 세로 스택으로 접고, 검색 입력을 가장 위에 둔다.
- 카드 레이아웃은 `제목 → 태그 → 가격/배송/상태 → 액션` 순으로 단순화한다.
- 상세 모달은 데스크톱에서는 중앙 패널, 모바일에서는 하단 시트에 가까운 큰 패널로 보이게 한다.

## 인증 및 데이터 저장 구조

### 인증
- Firebase Auth의 Google 로그인을 유지한다.
- 추가 이메일 입력은 받지 않는다.
- 로그인한 Google 계정의 이메일 주소를 메일 알림 기본 수신 주소로 사용한다.

### Firestore 저장 구조
- `userPreferences/{uid}`
  - `bookmarks: number[]`
  - `alertKeywords: string[]`
  - `emailAlertsEnabled: boolean`
  - `email: string`
  - `displayName: string`
  - `updatedAt`
- `alertDeliveries/{uid}_{dealHash}_{keywordHash}`
  - `uid`
  - `dealUrl`
  - `keyword`
  - `deliveredAt`
  - `title`
  - `email`

### 저장 원칙
- 스크랩과 키워드는 현재 로컬 저장과 Firestore 동기화를 유지한다.
- 메일 알림 사용 여부는 브라우저 알림과 분리해서 저장한다.
- 중복 발송 방지는 `uid + deal URL + keyword` 기준 기록으로 처리한다.

## 메일 알림 구조
- 수집 워크플로는 1시간마다 실행한다.
- 새 딜 저장 후 Firestore에서 메일 알림 사용자가 저장한 키워드를 불러온다.
- 새 딜 제목, 상품명, 요약, 태그, 플랫폼을 합친 검색 대상으로 키워드 매칭을 수행한다.
- 매칭된 사용자에게 즉시 메일을 발송한다.
- 메일 본문은 아래 정보만 짧게 보낸다.
  - 제목
  - 가격
  - 배송
  - 플랫폼
  - 구매 링크
  - 출처
  - 업데이트 시각
- 메일 발송 수단은 GitHub Actions 비밀값으로 구성한 SMTP 또는 메일 API를 사용한다.
- 워크플로 실패가 메인 수집 저장을 막지 않도록, 수집과 메일 발송 단계는 분리하고 발송 오류는 별도 로그로 남긴다.

## 수집 워크플로 변경
- 기존 3시간 주기 크론을 1시간 주기로 변경한다.
- 목표 크론 값은 `0 * * * *`
- 흐름은 다음 순서로 구성한다.
  1. 딜 수집
  2. 정규화 및 카테고리 분류
  3. `data/deals.json` 저장
  4. Git 커밋
  5. Firestore 구독 로드
  6. 키워드 매칭 메일 발송

## 오류 처리
- Firebase 설정이 없더라도 페이지는 읽기 전용으로 동작해야 한다.
- Google 로그인이 비활성화된 경우 로그인 버튼은 안내 문구 없이 비활성 또는 숨김 상태로 단순하게 처리한다.
- 메일 발송 설정이 누락되면 수집은 정상 완료하고 발송만 건너뛴다.
- 구매 링크를 찾지 못한 딜은 카드와 상세의 대표 구매 버튼을 숨긴다.
- 카테고리 분류에 실패한 딜은 `할인페스타`로 밀어 넣기보다 제목·플랫폼 힌트를 추가 검사한 뒤 최종 fallback을 적용한다.

## 테스트 전략
- 카테고리 분류 규칙 단위 테스트를 확장한다.
- 제목 정규화 `[플랫폼] 상품명` 테스트를 추가한다.
- 상태 표시 규칙 테스트를 추가한다.
  - deadline 있음
  - deadline 없음 + 선착순 있음
  - 둘 다 없음
- UI 테스트는 모바일 레이아웃과 홈 초기화 동작을 포함한다.
- 메일 매칭 로직은 샘플 딜과 키워드 조합으로 중복 발송 방지 테스트를 추가한다.

## 구현 순서
1. 카테고리 및 제목 정규화 규칙 정비

3. 모바일 레이아웃 개선

6. GitHub Actions 1시간 주기 및 메일 발송 단계 추가
7. 검증 및 운영 문서 갱신
