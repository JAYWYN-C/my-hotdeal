## Plan: 핫딜 사이트 MVP 구축 로드맵

자동 수집 기반의 한국형 핫딜 서비스 MVP를 먼저 웹/모바일웹으로 출시하고, 앱은 2차로 분리하는 접근입니다. 초기에는 수집 안정성·카테고리 품질·알림 신뢰도를 우선 확보하고, 이후 앱/고급 기능으로 확장합니다.

**Steps**
1. 요구사항 고정 및 IA 정의: 카테고리를 3개로 확정하고(B안), 각 카테고리의 태그 체계를 정의한다. *선행 단계*
2. 기술 스택/아키텍처 확정: 웹+모바일웹 우선에 맞춰 백엔드(수집/알림/API), 프론트(반응형), 배치/큐, DB를 선택한다. *depends on 1*
3. 데이터 모델 설계: 딜, 소스 사이트, 카테고리, 태그, 가격/기간, 작성자/수집메타, 사용자, 북마크, 알림구독, 알림이력을 정의한다. *depends on 2*
4. 수집 파이프라인 MVP 구축: 아카라이브/뽐뿌/네이버 카페·블로그/맘카페류를 대상으로 수집기 어댑터, 정규화, 중복제거, 실패재시도, 스케줄러를 만든다. *depends on 3*
5. 정책/컴플라이언스 반영: robots/약관/요청빈도/저작권 고지/원문링크 정책을 명문화하고 수집기별 제한을 설정한다. *parallel with 4, but blocks 6 release*
6. 사용자 기능 MVP 구현: 비회원 열람, Google OAuth 로그인, 스크랩, 키워드/카테고리/마감임박 알림 구독을 제공한다. *depends on 3*
7. 딜 탐색 UX 구현: 카테고리 홈, 목록/상세, 검색·필터(가격/마감/사이트), 정렬(최신/인기), 원문 이동 CTA를 구현한다. *parallel with 6, depends on 3*
8. 알림 엔진 구축: 조건 매칭, 이벤트 생성(신규/마감임박/종료), 채널(이메일 또는 푸시 대체) 발송, 중복 방지, 실패 재처리를 구현한다. *depends on 6 and 4*
9. 운영/관리 도구 구축: 관리자 검수, 신고/차단, 수집 상태 대시보드, 소스별 에러 모니터링을 만든다. *depends on 4 and 6*
10. 배포/관측성 설정: CI/CD, 로그·메트릭·알림, 백업, 장애 대응 런북을 구성하고 베타 오픈한다. *depends on 7,8,9*
11. 2차 확장(앱): API 안정화 후 모바일 앱을 추가하고, 웹 알림과 앱 푸시를 통합한다. *post-MVP*

**Relevant files**
- 현재 워크스페이스는 빈 상태이므로 재사용할 기존 코드 파일이 없다.
- 1단계에서 프로젝트 구조를 생성한 뒤, API/수집기/알림/인증/프론트 라우팅 기준 파일을 재정의한다.

**Verification**
1. 수집 검증: 소스별 샘플 URL 20개 기준 파싱 성공률/필드 완성도/중복률 측정
2. 카테고리 검증: 3개 분류 + 태그 정책에 대해 운영자 샘플 리뷰(오분류율 목표 설정)
3. 인증 검증: Google 로그인/세션 만료/재로그인/권한(비회원 vs 회원) 시나리오 테스트
4. 알림 검증: 신규 딜/마감임박/종료 이벤트의 지연시간, 중복발송, 실패 재시도 테스트
5. 성능 검증: 목록 API p95 응답시간, 동시 사용자 기준 부하 테스트
6. 운영 검증: 수집 실패 감지 알람, 관리자 검수 플로우, 장애 복구 리허설

**Decisions**
- 자동 수집(크롤링/피드) 포함
- 비회원 열람 + Google 로그인 회원 기능 제공(스크랩/알림)
- 출시 전략: 1차 웹+모바일웹, 2차 앱
- 수집 우선 소스: 아카라이브, 뽐뿌, 네이버 카페/블로그, 맘카페류
- 카테고리: 3개 분류(B안, 회사 행사 정보는 하위 태그 체계로 처리)

**Further Considerations**
1. 알림 채널 우선순위 권장: 이메일 먼저, 앱 푸시는 2차에서 도입
2. 법적 리스크 저감 권장: 원문 전문 저장보다 요약+원문 링크 중심 정책
3. 운영 초기 권장: 완전 자동 노출 대신 관리자 검수 큐를 거치는 하이브리드 공개
